That URL is Paymob’s secure callback endpoint, not Shopify’s:

https://accept.paymobsolutions.com/api/acceptance/mpgs_secure_callback/.../authorization_v2

So the error is almost certainly coming from your own backend logic that runs when Paymob calls back, not from Shopify and not from the Shopify APIs.

Shopify never calls that URL, and Shopify’s REST Order JSON has no shopify_payment field (Order REST 2024-04 schema). So the only way you see:

Order has no shopify_payment
during a Paymob callback is:

Somewhere in your callback handler code, you try to access a property like order.shopify_payment or similar when you sync the Paymob data with Shopify.
Because Paymob itself and its API are outside Shopify, I can’t debug Paymob’s internal behavior. But I can outline how to structure your Shopify side so that this error disappears and your sync is clean.

What’s really happening

Your flow is:

Customer pays via Paymob (3DS, capture, etc.).
Paymob hits your configured callback/webhook endpoint (or you poll) – in your example, via that mpgs_secure_callback/authorization_v2.
Your code in that callback handler probably:
Fetches or creates a Shopify DraftOrder.
Completes the draft.
Or reads an existing Shopify Order.
During that step, your code is doing something like:

How to fix it on your side (and stay in sync with Paymob)

1. Remove any usage of shopify_payment in your callback handler

In the code that handles the Paymob callback (the thing behind that mpgs_secure_callback/authorization_v2 URL):

Search for shopify_payment in the project.
Remove or replace any access like:
order.shopify_payment
payload.shopify_payment
or any validation rule that expects that field.
Instead, use fields your app owns, for example:

paymob_transaction_id
paymob_order_id
paymob_status
These values come from Paymob’s payload, not Shopify.

2. Use Shopify as an order record only; Paymob as payment record

Your sync pattern should look like this:

Create Shopify DraftOrder
POST /admin/api/2024-04/draft_orders.json

Use Paymob, not Shopify, for payment status & events

To answer “Is this paid?” → check Paymob’s status (e.g. captured).
To answer “Show me payment history?” → query Paymob /api/acceptance/transactions.
Shopify is simply an order and fulfillment record; the order only exists as “paid” because you completed the draft after Paymob confirmed.
Summary: what you must change to “not have this error”

The error is not from Shopify; it’s from your code (or a library) that expects shopify_payment on a Shopify order.
In the Paymob callback handler, remove any reference to shopify_payment and replace it with:
Fields that come from Paymob (transaction_id, status), and
Optionally custom fields you store on the Shopify order (paymob_transaction_id in note_attributes or metafields).
Keep using the standard Shopify REST resources (DraftOrder, Order) exactly as documented:
DraftOrder – REST 2024-04
Order – REST 2024-04
