FINAL ADDITIONS / EDGE CASES (MUST APPEND TO PLAN)

1. PAYMOB IFRAME + 3DS / AUTHENTICATION EDGE CASES

Some Paymob payment flows (especially card payments) may trigger:
- 3D Secure verification pages
- bank OTP screens
- external authentication redirects

These flows may:
- break iframe containment
- open new tabs
- temporarily exit iframe context

Rule:
The system must NOT assume iframe persistence throughout payment.
If Paymob exits iframe context during authentication, the system must still recover via backend verification and/or fallback redirect without breaking the checkout flow.

---

2. POSTMESSAGE EVENT FILTERING (STRICT)

Paymob postMessage events are NOT all payment confirmations.

The system must ONLY react to explicitly documented success events.

Rules:
- Ignore all non-payment events (resize, navigation, pending, etc.)
- Only trigger verification on confirmed "payment success" event type
- Never assume any generic "success" message = paid

---

3. CONCURRENT VERIFY RACE CONDITION PROTECTION

Even with idempotency, simultaneous calls to /paymob-verify can occur.

To prevent double processing:
- Implement atomic state transition in DB:
  pending → processing → paid

Rules:
- First request locks intentId into "processing"
- Other requests must wait or short-circuit
- Only one request is allowed to call Paymob inquiry API per intent at a time

---

4. CART + CHECKOUT STATE PERSISTENCE (RELOAD SAFETY)

Checkout state must survive:
- page refresh
- iframe fallback redirect
- mobile Safari reload
- accidental tab close/reopen

Solution:
- persist cart + checkout state in localStorage or sessionStorage
- optionally sync to backend session

Rule:
No user should lose cart or address data due to reload during payment flow.

---

5. IFRAME TIMEOUT RULE (10s MUST BE NON-DESTRUCTIVE)

If iframe does not load within 10 seconds:
- trigger fallback redirect

BUT:
- if iframe starts loading after timeout was scheduled, cancel fallback
- do NOT interrupt active payment rendering

---

6. APPLE PAY CLARIFICATION (CAPABILITY ONLY)

Apple Pay visibility only determines UI rendering.

It does NOT guarantee:
- successful payment
- correct merchant setup
- Paymob Apple Pay activation

Final payment success must always be confirmed via backend verification.

---

7. SHOPIFY ORDER SINGLE SOURCE OF TRUTH

Shopify order creation MUST ONLY happen in backend:

processPaymobSuccess()

Frontend MUST NEVER:
- create orders
- infer payment success
- mutate order state

Backend is the only authority for order creation.

---

8. VERIFY FAILURE RETRY LOGIC

If /paymob-verify fails due to:
- network issues
- Paymob API downtime
- timeout

Then:
- DO NOT clear cart
- DO NOT reset checkout
- allow user to retry verification without re-paying

---

9. CART DRAWER UX ENHANCEMENT (ADD THIS)

Cart drawer bottom section must also display structured checkout summary:

- Contact
- Ship to
- Method
- Total

This summary should appear above the payment buttons and reflect current checkout state.

Example layout:

Contact:       phone/email
Ship to:       address + governorate
Method:        Paymob / Apple Pay
Total:         899 EGP

This ensures transparency before payment and reduces checkout drop-off.

---

10. FINAL SYSTEM PRINCIPLE

If any conflict exists:

BACKEND PAYMENT STATE > PAYMOB RESPONSE > FRONTEND UI STATE

Frontend is always a reflection layer, never a source of truth.
